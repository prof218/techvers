# Uber Clone Architecture Case Study

This case study reviews the system design, geofencing mechanisms, database schemas, and WebSocket dispatch pipelines required to construct a reliable, real-time ride-matching platform.

## System Architecture

Ride-sharing applications are built on a high-availability event-driven architecture, connecting drivers and riders dynamically based on coordinate matches.

```
+---------------+              +---------------+
|  Rider Client | <-----------> |               |
+---------------+              |   WebSocket   |
                               | Gateway Server|
+---------------+              |               |
| Driver Client | <-----------> |               |
+---------------+              +---------------+
                                       |
                                       v
                               +---------------+      +--------------+
                               | Dispatcher    | ---> | Kafka Topic  |
                               | (Node.js)     |      | (Location)   |
                               +---------------+      +--------------+
```

### Core Architecture Components
1. **Websocket Gateways** — Manages persistent bi-directional communication channels for both drivers and riders. Receives driver location pings every 4 seconds.
2. **Geospatial Indexer** — Maps coordinate pairs (lat, lng) to hexagonal H3 cells. Restricts matching searches to adjacent driver cells.
3. **Dispatch Coordinator** — Queries nearby idle drivers, scores compatibility, sends push offers, and processes driver acceptances.

## Database Design

Rider information and billing records require high integrity and are stored in PostgreSQL. Real-time locations are held exclusively in memory within a Redis cluster.

### Driver Location Key (Redis GEO)
- Drivers post location data to a Redis Geo spatial index:
```bash
# Add driver location (lng, lat, driverId)
GEOADD drivers:active -122.4194 37.7749 driver_998312
```

### Ride Record Schema (PostgreSQL Table: `rides`)
```sql
CREATE TABLE rides (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rider_id UUID REFERENCES users(id),
    driver_id UUID REFERENCES users(id),
    status VARCHAR(50) DEFAULT 'requested', -- requested, accepted, active, completed, cancelled
    pickup_coords POINT NOT NULL,
    dropoff_coords POINT NOT NULL,
    fare_cents INTEGER NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP WITH TIME ZONE
);

CREATE INDEX idx_rides_status ON rides(status);
```

## Scalability & Performance

### 1. GPS Ping Throttle & Buffering
Sending driver GPS updates continuously can overwhelm servers. Driver clients compress coordinates, batch them, and send updates every 4 seconds. Gateways write these coordinates directly to a Kafka location stream, decoupling ingestion from matching operations.

### 2. Localized Geo-Hashing
Instead of scanning all drivers, driver searches are bucketed using Uber's H3 Hexagonal Hierarchical Index. This ensures search operations have an $O(1)$ lookup complexity, returning nearby driver ids in under **5ms**.
